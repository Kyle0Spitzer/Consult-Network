Rendimiento, seguridad y alta disponibilidad para arquitecturas híbridas, multi-cloud y cross-cloud.

Arquitectura de referencia

Arquitectura cloud networking para conectar, proteger y escalar entornos híbridos, multi-cloud y cross-cloud

La arquitectura se estructura en tres capas: un punto de entrada global, una interconexión neutral híbrida y Landing Zones nativas en cada hyperscaler

01

Punto de entrada global

Front door unificado para tráfico corporativo, clientes de Internet y sedes SD-WAN, con Anycast para dirigir cada petición al endpoint más eficiente.

02

Interconexión neutral y hubs de conectividad

Hub híbrido central para conectar hyperscalers y CPDs corporativos mediante circuitos dedicados, fabric, MPLS, ISP, P2P o BYOP, sin caer en una topología full mesh difícil de gobernar.

03

Landing zones por cloud

Arquitectura hub-and-spoke, segmentación por entornos y servicios nativos para routing, balanceo, firewall, DNS y conectividad privada hacia PaaS.

Principio de diseño

Cloud-centric y asset-light

La red se diseña desde la aplicación, los servicios cloud y sus patrones de consumo, reduciendo dependencias de hardware dedicado y priorizando capacidades nativas cuando aportan soporte, evolución y operación más simple.

Principio de diseño

Transporte y overlay separados

La conectividad física, los fabrics y los carriers se desacoplan del overlay lógico de routing, segmentación y seguridad. Esto permite cambiar transporte sin rediseñar la arquitectura de red consumida por las aplicaciones.

Principio de diseño

Escalabilidad modular

Las capacidades se publican como patrones componibles para crecer por dominio, cloud, región o criticidad. El objetivo es evolucionar sin reconstruir la plataforma ni crear excepciones difíciles de operar.

Principio de diseño

Neutralidad tecnológica

El diseño evita acoplarse innecesariamente a un carrier, fabricante o cloud concreta. La consistencia se consigue con patrones comunes, mientras cada proveedor mantiene sus capacidades nativas.

Principio de diseño

Visibilidad end-to-end

La operación necesita telemetría de salud, rendimiento, capacidad, costes y flujos para tomar decisiones con contexto. La observabilidad debe cubrir edge, hubs, clouds, conectividad privada y servicios compartidos.

Principio de diseño

Control y Zero Trust

El acceso se gobierna por identidad, contexto, postura y segmentación, con exposición mínima y conectividad privada por defecto cuando aplica. Las políticas deben acompañar al servicio aunque cambie de cloud o dominio.

Principio de diseño

Resiliencia por criticidad

La redundancia local, geográfica y la diversidad física se dimensionan según impacto real de negocio. No todos los servicios necesitan el mismo nivel de resiliencia, pero todos deben tener una decisión explícita y trazable.

Principio de diseño

Network as Code

Los cambios de red se modelan como código con Terraform, GitOps, validaciones y revisión trazable. Esto hace reproducible la evolución de Landing Zones, reduce riesgo operativo y facilita auditoría.

Beneficios clave

Reference architecture

Cloud networking architecture to connect, protect and scale hybrid, multi-cloud and cross-cloud environments

The architecture is structured in three layers: a global entry point, a neutral hybrid interconnection layer and native Landing Zones in each hyperscaler

01

Global entry point

Unified front door for corporate traffic, Internet customers and SD-WAN sites, using Anycast to route each request to the most efficient endpoint.

02

Neutral interconnection and connectivity hubs

Central hybrid hub to connect hyperscalers and corporate data centers through dedicated circuits, fabric, MPLS, ISP, P2P or BYOP, without falling into a full-mesh topology that is hard to govern.

03

Cloud Landing Zones

Hub-and-spoke architecture, environment segmentation and native services for routing, load balancing, firewall, DNS and private connectivity to PaaS.

Design principle

Cloud-centric and asset-light

The network is designed from the application, cloud services and their consumption patterns, reducing dependency on dedicated hardware and prioritizing native capabilities when they improve support, evolution and operations.

Design principle

Separate transport and overlay

Physical connectivity, fabrics and carriers are decoupled from the logical overlay for routing, segmentation and security. This allows transport changes without redesigning the network architecture consumed by applications.

Design principle

Modular scalability

Capabilities are published as composable patterns that can grow by domain, cloud, region or criticality. The goal is to evolve without rebuilding the platform or creating exceptions that are hard to operate.

Design principle

Technology neutrality

The design avoids unnecessary coupling to a specific carrier, vendor or cloud. Consistency comes from common patterns, while each provider keeps its native capabilities.

Design principle

End-to-end visibility

Operations need telemetry for health, performance, capacity, costs and flows to make decisions with context. Observability should cover edge, hubs, clouds, private connectivity and shared services.

Design principle

Control and Zero Trust

Access is governed by identity, context, posture and segmentation, with minimum exposure and private connectivity by default where applicable. Policies should follow the service even when it moves across clouds or domains.

Design principle

Resilience by Criticality

Local redundancy, geographic redundancy and physical diversity are sized according to real business impact. Not every service needs the same resilience level, but every service should have an explicit, traceable decision.

Design principle

Network as Code

Network changes are modeled as code with Terraform, GitOps, validations and traceable review. This makes Landing Zone evolution reproducible, reduces operational risk and supports auditability.

Key benefits

Design principles

A cloud-first, modular and end-to-end operable architecture

The model avoids a proprietary multi-cloud abstraction layer. Each cloud uses its native capabilities, while the platform publishes common patterns to consume networking consistently, with governance and automation.
